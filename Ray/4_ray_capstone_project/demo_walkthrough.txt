Demo Walkthrough — Ray Capstone: TLC-backed per-zone recommendations under skew
=================================================================================

SETUP (before recording)
- conda activate 22971-ray-capstone
- bash reset.sh
- Download data: the test_flow.sh script auto-downloads from TLC if not present
- Have two terminals open: one for running, one for inspecting artifacts

PART 1 — Code Walkthrough (~3 min)
- Show project files: tlc.py, prepare.py, run.py
- tlc.py:
  - Constants, dataclasses (RunConfig, ZoneSnapshot, ZoneDecision, TickMetrics)
  - Data loading and validation (load_parquet, validate_adjacent_months)
  - Zone selection: select_active_zones — top-N busiest, deterministic with seed
  - Baseline building: aggregate_ticks → build_baseline_table (zone, hour, dow)
  - Scoring rule: compute_decision — NEED if recent_avg > baseline_mean + 1 std
  - Fallback: apply_fallback — always_previous policy, OK for first tick
- prepare.py:
  - Loads 2 adjacent parquets, validates months, selects zones, writes prepared assets
  - Cross-check: confirms replay counts match direct pandas calculation
- run.py:
  - ZoneActor: one per zone, owns replay partition + baseline + mutable state
    - activate_tick, get_snapshot, report_decision, write_decision, finalize_tick
    - Idempotent writes by (zone_id, tick_id)
    - Counters for duplicates, late reports, fallbacks
  - score_zone: Ray remote task, receives snapshot, returns decision
    - Blocking: returns to controller
    - Async: reports to actor via actor_handle
  - Blocking driver: launch all tasks → ray.get all → write decisions
  - Async driver: bounded concurrency (max_inflight_zones), ray.wait with timeout,
    polls actor readiness, finalizes with fallback policy
  - Stress driver: reuses both with harsher skew (60% slow, 3s sleep)

PART 2 — Blocking Baseline Run (~2 min)
- Run: python run.py --prepared-dir prepared --output-dir output --mode blocking --slow-zone-fraction 0.25 --slow-zone-sleep-s 1.0
- Show logs: tick latency dominated by slowest zones
- Open output/blocking/metrics.csv: point out max_zone_latency vs mean
- Open output/blocking/tick_summary.json: show per-zone decisions
- Key point: every tick waits for all zones, skew directly hurts latency

PART 3 — Async Controller Run (~2 min)
- Run: python run.py --prepared-dir prepared --output-dir output --mode async --tick-timeout-s 2.0
- Show logs: tick finalization after timeout, fallback applied to late zones
- Open output/async/metrics.csv: lower total tick latency
- Open output/async/actor_counters.json: show late/duplicate/fallback counts
- Key point: tick completes without waiting for stragglers, fallback used

PART 4 — Stress Test (~2 min)
- Run: python run.py --prepared-dir prepared --output-dir output --mode stress
- Show output/stress/comparison.json: blocking vs async under 60% slow zones
- Blocking: much higher tick latency
- Async: controlled tick completion, more fallbacks but ticks finish predictably
- Key point: async degrades gracefully, blocking degrades sharply

PART 5 — Summary (~1 min)
- Recap: 3 runs on same replay data with same scoring logic
- Actor-owned state with idempotent writes
- Blocking: simple but skew-sensitive
- Async: bounded concurrency + timeout + fallback = controlled degradation
- Artifacts: run_config, metrics, latency_log, tick_summary, actor_counters
- All invariants: idempotent writes, no double-counting, late reports logged and ignored
